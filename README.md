## Возможности

- регистрация и аутентификация пользователей по JWT;
- безопасное хеширование паролей с помощью BCrypt;
- создание временного гостевого профиля;
- получение и редактирование пользовательского профиля;
- рейтинги, лиги, статистика и история матчей;
- лобби подключённых игроков в реальном времени;
- отправка, принятие и отклонение приглашений;
- создание игровых сессий для двух игроков;
- серверная проверка и применение игровых ходов;
- персональная рассылка состояния матча каждому участнику;
- серверный таймер хода с периодическими обновлениями;
- сохранение результатов матчей и изменение рейтинга;
- управление схемой базы данных через Liquibase;
- интерактивная документация REST API через Swagger UI.

## Архитектура

Сервер организован по принципам многослойной архитектуры:
каждый слой имеет отдельную ответственность и взаимодействует 
с нижележащими компонентами.

```text
┌─────────────────────────────────────────────────┐
│ Controller / WebSocket                          │
│ REST API и обработка STOMP-сообщений            │
├─────────────────────────────────────────────────┤
│ Service / Game Core                             │
│ Бизнес-логика, игровой движок и управление игрой│
├─────────────────────────────────────────────────┤
│ Repository                                      │
│ JPA-репозитории и in-memory хранилища           │
├─────────────────────────────────────────────────┤
│ Entity / Domain Model                           │
│ Сущности базы данных и модели состояния игры    │
├─────────────────────────────────────────────────┤
│ PostgreSQL / H2 / In-memory                     │
│ Постоянное и временное хранение данных          │
└─────────────────────────────────────────────────┘
```

REST API используется для работы с аккаунтами, пользовательскими данными
и статистикой. STOMP поверх WebSocket обслуживает лобби, приглашения,
игровые события и таймер.

```text
Android-клиент
├── REST API
│   └── Сервисы приложения
│       └── PostgreSQL / H2
└── STOMP / WebSocket
    ├── Лобби и приглашения
    │   └── In-memory хранилище
    └── Игровой движок
        ├── Активные матчи
        │   └── In-memory хранилище
        └── Завершённые матчи
            └── PostgreSQL / H2
```

Основные компоненты:

- `controller` — REST- и STOMP-контроллеры;
- `service` — бизнес-логика пользователей, лобби и приглашений;
- `game/core` — игровой движок, игровой цикл, матчи и таймер;
- `game/effect` — реализации эффектов игровых карт;
- `game/model` — модели состояния матча, игроков, карт и кубиков;
- `repository` — JPA-репозитории и временные in-memory хранилища;
- `entity/db` — сущности базы данных;
- `listener` — обработчики WebSocket-подключений и отключений;
- `jwt` и `config` — аутентификация, безопасность и конфигурация;
- `db.changelog` — миграции и начальные данные Liquibase.

### Жизненный цикл матча

1. Игрок подключается к WebSocket и появляется в лобби.
2. Один игрок отправляет другому приглашение.
3. После принятия приглашения сервер создаёт игровую сессию и отправляет
   обоим участникам `gameId`.
4. Оба клиента подтверждают готовность.
5. Сервер отправляет начальное состояние матча.
6. Клиент отправляет ход, после чего игровой движок проверяет и применяет его.
7. Обновлённое состояние и таймер рассылаются обоим участникам.
8. После победы или сдачи результат сохраняется в базе данных.

## Безопасность

Защита HTTP API реализована с помощью Spring Security. Сервер использует
stateless-аутентификацию: состояние HTTP-сессии между запросами не хранится.

```text
HTTP-запрос
└── JwtAuthenticationFilter
    ├── Извлечение Bearer-токена
    ├── Проверка подписи и срока действия
    ├── Загрузка пользователя и его полномочий
    └── Передача запроса в контроллер
```

Основные механизмы безопасности:

- пароли хешируются с помощью BCrypt;
- JWT подписывается секретным HMAC-ключом;
- в `subject` токена сохраняется email пользователя;
- срок действия токена задаётся параметром `JWT_EXPIRATION`;
- `JwtAuthenticationFilter` выполняется перед стандартным
  `UsernamePasswordAuthenticationFilter`;
- пользовательские роли хранятся в базе данных;
- HTTP-сессии отключены через `SessionCreationPolicy.STATELESS`;
- все маршруты, не объявленные публичными, требуют аутентификации;
- CSRF отключён, поскольку REST API использует Bearer-токены и не опирается
  на cookie-based HTTP-сессии.

## Хранение данных

Проект использует постоянное и временное хранение данных.

| Хранилище | Содержимое |
| --- | --- |
| PostgreSQL | Пользователи, роли, лиги, достижения, результаты |
| H2 in-memory | Локальная версия постоянной модели данных |
| Память приложения | Лобби, приглашения, готовность и активные матчи |

PostgreSQL используется в удалённом окружении. H2 позволяет запускать сервер
локально без установки отдельной СУБД для быстрого тестирования.

Схема базы данных управляется Liquibase. При старте приложения Liquibase:

1. проверяет применённые changeset;
2. создаёт отсутствующие таблицы и связи;
3. применяет новые миграции;
4. загружает предусмотренные проектом начальные данные.

## Конфигурация

Приложение поддерживает три Spring-профиля.

| Профиль | Назначение | База данных |
| --- | --- | --- |
| `local` | Локальная разработка | H2 in-memory |
| `docker` | Контейнерное окружение | PostgreSQL |
| `remote` | Работа удалённого сервера | PostgreSQL |

Основные настройки профилей находятся в `application.yml`. Секретные
параметры и реквизиты подключения к базе данных хранятся во внешних
profile-specific файлах:

- `application-local.properties` — локальная конфигурация;
- `application-remote.properties` — конфигурация удалённого сервера.

> [!WARNING]
> Файлы `application-local.properties` и `application-remote.properties`
> добавлены в `.gitignore` и существуют только в соответствующем окружении.

### Локальная конфигурация

Файл `application-local.properties` размещается в корне проекта
и содержит настройки JWT:


Параметры подключения к H2 уже определены для профиля `local` в

## REST API

За исключением явно публичных маршрутов, запросы должны содержать JWT:

```http
Authorization: Bearer <JWT_TOKEN>
```

### Авторизация

| Метод | Маршрут | Доступ | Назначение |
| --- | --- | --- | --- |
| `POST` | `/api/auth/register` | Публичный | Регистрация пользователя |
| `POST` | `/api/auth/login` | Публичный | Авторизация и получение JWT |

Пример регистрации:

```bash
curl -X POST http://localhost:8080/api/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"email":"player@example.com","password":"strong-password"}'
```

Пример авторизации:

```bash
curl -X POST http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"player@example.com","password":"strong-password"}'
```

Успешная авторизация возвращает JWT и данные пользователя:

```json
{
  "token": "<JWT_TOKEN>",
  "user": {
    "id": 1,
    "email": "player@example.com",
    "league": "...",
    "nickname": "...",
    "ratingPoints": 0,
    "winsAmount": 0,
    "gamesPlayed": 0,
    "createdAt": "..."
  }
}
```

### Пользователи, гости, история

| Метод | Маршрут | Доступ | Назначение |
| --- | --- | --- | --- |
| `POST` | `/api/guest/create` | Публичный | Создать гостевой ID |
| `GET` | `/api/users` | JWT | Получить пользователей |
| `GET` | `/api/users/{id}` | JWT | Получить пользователя |
| `GET` | `/api/users/me` | JWT | Получить текущий профиль |
| `PUT` | `/api/users` | JWT | Обновить профиль |
| `DELETE` | `/api/users/{id}` | JWT | Удалить пользователя |
| `GET` | `/api/history/{userId}` | JWT | Получить историю матчей |

`POST /api/guest/create` возвращает временный идентификатор вида
`guest-<UUID>`. Он используется для подключения неавторизованного игрока
к лобби и не создаёт постоянную учётную запись.

## WebSocket и STOMP

WebSocket используется для двустороннего обмена игровыми событиями без
периодического опроса сервера.

### Точки подключения

| Endpoint | Назначение |
| --- | --- |
| `/ws` | WebSocket endpoint с поддержкой SockJS |
| `/ws-android` | Нативный WebSocket для Android-клиента |

Android-клиент передаёт идентификатор пользователя во время handshake:

```text
ws://localhost:8080/ws-android?userId=<USER_ID>
```

Для гостя используется идентификатор, полученный от
`POST /api/guest/create`.

Конфигурация STOMP:

| Префикс | Назначение |
| --- | --- |
| `/app` | Сообщения от клиента к контроллерам |
| `/topic` | Общие broadcast-события |
| `/queue` | Очереди сообщений |
| `/user` | Персональные сообщения пользователя |

### Сообщения от клиента к серверу

| Destination | Payload | Назначение |
| --- | --- | --- |
| `/app/ping` | Без payload | Проверить соединение |
| `/app/lobby` | Без payload | Запросить лобби |
| `/app/invitations` | Без payload | Получить приглашения |
| `/app/invite` | `InvitationDTO` | Отправить приглашение |
| `/app/invite.reject` | `InvitationDTO` | Отклонить приглашение |
| `/app/invite.accept` | `InvitationDTO` | Принять приглашение |
| `/app/game/{gameId}/ready` | Без payload | Подтвердить готовность |
| `/app/game/{gameId}/move` | `MoveDto` | Выполнить игровой ход |

Пример приглашения:

```json
{
  "senderId": "1",
  "senderNickname": "Player One",
  "receiverId": "2",
  "sendTime": "2026-07-20T12:00:00Z"
}
```

Каждый игровой ход содержит поле `type`. Поддерживаемые значения:

| `type` | Назначение |
| --- | --- |
| `PLAY_CARD` | Применить эффект карты к слоту |
| `ROTATE_DICE` | Изменить состояние кубика |
| `SWAP_DICES` | Поменять кубики местами |
| `DOUBLE_TAP` | Применить эффект двойного хода |
| `RESHUFFLE_CARD` | Заменить выбранные карты в руке |
| `SURRENDER` | Сдаться |

Пример хода:

```json
{
  "type": "PLAY_CARD",
  "playerId": "1",
  "cardId": "card-uuid",
  "targetSlotIndex": 0,
  "targetPlayerId": "PLAYER"
}
```

Для обозначения владельца слота используются значения `PLAYER` и `OPPONENT`.
Состояния кубика: `ZERO`, `ONE`, `PLUS`, `MINUS`, `I`, `I_MINUS`.

### Подписки клиента

| Destination | Payload | Событие |
| --- | --- | --- |
| `/topic/lobby` | `LobbyResponseDTO` | Обновление лобби |
| `/user/queue/pong` | timestamp | Ответ на ping |
| `/user/queue/invitations` | `InvitationEventDTO` | Событие приглашения |
| `/user/queue/game.start` | `GameStartEventDto` | Получение `gameId` |
| `/user/queue/game/{gameId}` | `GameState` | Состояние матча |
| `/user/queue/game/{gameId}/timer` | `TimerUpdatePacket` | Таймер хода |

Клиент подписывается на адреса `/user/queue/...`. Spring самостоятельно
маршрутизирует сообщение в WebSocket-сессию текущего пользователя.

## Игровой процесс

Игровой движок является сервер-авторитетным: клиент отправляет намерение
совершить ход, а сервер проверяет игрока, очередь хода, наличие карты,
совместимость эффекта и состояние выбранного слота.

В текущей реализации:

- в матче участвуют два игрока;
- каждому игроку выдаётся рука до 6 карт;
- у каждого игрока создаются 4 слота с кубиками;
- длительность хода составляет 45 секунд;
- обновление таймера выполняется каждые 500 миллисекунд;
- победитель определяется при совпадении текущих и требуемых состояний всех
  его кубиков;
- зарегистрированным игрокам после матча обновляются статистика и рейтинг;
- гости могут участвовать в матчах, но для них не сохраняется статистика.
